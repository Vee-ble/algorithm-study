### **1. 요구사항 분석**

**요구사항 요약:**

1. 임의의 한 위치부터 k개를 연속해서 먹으면 할인된 정액 가격으로 제공
2. 각 고객에게 초밥 종류 하나가 쓰인 쿠폰을 발행, 1번 행사에 참여하면 이 쿠폰에 적혀진 초밥 하나 무료 제공. 만약 이 번호에 적힌 초밥이 벨트 위에 없으면 새로 만들어서 제공.

초밥 가짓수의 최대값을 구하라.

**제약 사항(Constraints): 2 ≤ N ≤ 30,000, 2 ≤ d ≤ 3,000, 2 ≤ k ≤ 3,000 (k ≤ N), 1 ≤ c ≤ d**

**예외 케이스(Edge Case):**

- N이 최대값일 경우,

0 1 2 3

1 2 3 4

### **2. 해결 전략 및 자료구조 설계**

- **해결 전략 (Algorithm):**
  1. 길이 `N`의 1차원 배열에 초밥 정보를 저장
  2. 크기가 `d + 1`인 `count` 배열을 만들어, 현재 윈도우(연속된 `k`개) 안에 들어있는 초밥 종류별 개수를 기록한다.
  3. **초기 세팅:** 인덱스 `0`부터 `k-1`까지 최초의 윈도우를 구성하고, `count` 배열을 채우며 현재 먹을 수 있는 초밥 가짓수(`currentTypes`)를 구한다.
  4. **슬라이딩 윈도우:** 윈도우를 한 칸씩 오른쪽으로 이동시킨다 (총 `N`번 반복).
     - **빠지는 초밥(왼쪽 끝):** 현재 윈도우의 맨 앞 초밥 개수를 `count`에서 1 뺀다. 만약 개수가 0이 되었다면 `currentTypes`를 1 감소시킨다.
     - **들어오는 초밥(오른쪽 끝):** 새롭게 윈도우에 들어올 초밥(인덱스는 `(i + k) % N`으로 원형 처리)의 개수가 `count`에서 0이었다면 `currentTypes`를 1 증가시킨다. 그리고 개수를 1 더한다.
  5. 매 윈도우마다, 쿠폰 번호 `c`의 초밥 개수가 `count` 배열에서 0이라면 `currentTypes + 1`을 임시 최댓값으로 산정하여 최종 최댓값(`maxTypes`)을 갱신한다.
- **선택한 자료구조 근거:**
  - **선택:** 크기가 `d + 1`인 정수형 카운트 배열 (JS에서는 `Int32Array` 권장).
  - **이유:** `Set`을 사용하면 윈도우 안에 중복된 초밥이 여러 개 있을 때(예: 새우, 새우, 연어), 맨 앞의 '새우'가 윈도우를 빠져나가더라도 여전히 윈도우 안에 '새우'가 1개 남아있다는 사실을 추적할 수 없습니다. 따라서 **각 초밥의 개수를 정확히 세어주는 배열**이 필수적입니다.
- **예상 시간/공간 복잡도:** \* 시간 복잡도: O(N). 최초 `k`개를 탐색하고, 이후 `N`번 윈도우를 한 칸씩 이동할 때마다 양끝 요소 2개만 갱신(O(1))하므로 매우 빠릅니다.
  - 공간 복잡도: O(N + d). 초밥 벨트 배열 O(N)과 카운트 배열 O(d).

### 3. 의사코드 & 검증

```jsx
// N: 접시 수, d: 가짓수, k: 연속해서 먹는 수, c: 쿠폰 번호, sushi: 초밥 배열
count = 크기가 d+1인 0으로 초기화된 배열
currentTypes = 0

// 1. 초기 윈도우 (0번 ~ k-1번) 세팅
for i from 0 to k-1:
    if count[sushi[i]] == 0:
        currentTypes++
    count[sushi[i]]++

// 초기 윈도우 상태에서의 최댓값 세팅
maxTypes = currentTypes
if count[c] == 0:
    maxTypes = currentTypes + 1

// 2. 슬라이딩 윈도우 (한 칸씩 밀며 확인)
for i from 0 to N-1:
    // 앞쪽 초밥 제거
    leftSushi = sushi[i]
    count[leftSushi]--
    if count[leftSushi] == 0:
        currentTypes--

    // 뒤쪽 초밥 추가 (원형 배열 처리)
    rightSushi = sushi[(i + k) % N]
    if count[rightSushi] == 0:
        currentTypes++
    count[rightSushi]++

    // 쿠폰 적용 가짓수 확인
    currentMax = currentTypes
    if count[c] == 0:
        currentMax++

    // 최댓값 갱신
    if currentMax > maxTypes:
        maxTypes = currentMax

print maxTypes
```

### **4. 최종 구현 코드**

```jsx
const fs = require("fs");
const path = require("path");

const filePath =
  process.platform === "linux"
    ? "/dev/stdin"
    : path.join(__dirname, "input.txt");
const input = fs.readFileSync(filePath, "utf-8").trim().split(/\r?\n/);

const [N, d, k, c] = input[0].split(" ").map(Number);
const sushi = new Int32Array(N);
for (let i = 1; i <= N; i++) {
  sushi[i - 1] = Number(input[i]);
}

const count = new Int32Array(d + 1);
let currentTypes = 0;

// 처음 k개의 초밥으로 초기 윈도우 세팅
for (let i = 0; i < k; i++) {
  if (count[sushi[i]] === 0) {
    currentTypes++; // 처음 먹어본 스시라면 가짓수 증가.
  }
  count[sushi[i]]++;
}

// 초기 윈도우 상태에서의 최댓값 계산 (쿠폰 적용)
let maxTypes = currentTypes;
if (count[c] === 0) {
  maxTypes = currentTypes + 1;
}

// 슬라이딩 윈도우 시작
for (let i = 0; i < N; i++) {
  // 창문 맨 왼쪽에 있던 초밥은 창문에서 벗어남. 초밥 개수 1개 감소.
  const leftSushi = sushi[i];
  count[leftSushi]--;
  if (count[leftSushi] === 0) {
    // 개수를 줄였는데 0이라면, 가짓수도 감수
    currentTypes--;
  }

  // 창문이 이동했으니 창문의 오른쪽 끝에 초밥이 들어옴.
  // i + k는 새로 들어올 초밥의 인덱스. 레일은 둥글기에 배열의 끝을 넘어가면 0번으로 돌아와 야함.
  const rightSushi = sushi[(i + k) % N];
  if (count[rightSushi] === 0) {
    currentTypes++; // 초밥의 개수가 0이 었다면 가짓수 증가
  }
  count[rightSushi]++;

  let currentMax = currentTypes;
  if (count[c] === 0) {
    currentMax++; // 쿠폰 사용
  }

  // 현재 창문에서의 가짓수 > 역대 최고 가짓수 라면 기록 경신
  if (currentMax > maxTypes) {
    maxTypes = currentMax;
  }
}

console.log(maxTypes);
```

### 5. 트러블 슈팅 & 회고 (Retrospective)

**🔥 발생한 문제 (Trouble)**

- **논리적 오류 (중복 처리 및 조기 종료):** 초기 코드에서는 연속된 `k`개 안에 중복된 초밥이 나오면 탐색을 포기하고 포인터를 초기화했습니다. 또한, 쿠폰 번호가 없는 `k`개의 묶음을 찾으면 최댓값이라고 단정 짓고 `break`로 반복문을 완전히 종료해버려, 뒤에 있을 수 있는 더 나은 경우의 수를 놓치는 문제가 있었습니다.
- **자료구조의 한계:** `Set`을 사용하여 초밥의 종류를 파악하려 했으나, `Set`은 중복을 허용하지 않아 윈도우 안에 같은 초밥이 여러 개 있을 때(예: 연어 2개) 그중 하나가 윈도우를 빠져나가더라도 남은 연어의 개수를 추적할 수 없었습니다.
- **원형 배열 미처리:** 초밥 레일은 둥글게 이어져 있는데, 기존 반복문은 배열의 끝(`N - 1`)에서 탐색을 멈춰 배열의 끝과 처음이 이어지는 구간을 검사하지 못했습니다.

**🎯 해결 과정 (Shooting)**

- **카운팅 배열 도입:** `Set` 대신 초밥의 가짓수(`d + 1`)만큼의 크기를 가진 정수형 배열(`count`)을 선언했습니다. 이를 통해 현재 윈도우 안에 특정 초밥이 '몇 개' 들어있는지 정확히 추적할 수 있게 되었습니다.
- **슬라이딩 윈도우 적용:** 매번 `k`개를 처음부터 다시 세는 것이 아니라, 윈도우를 한 칸씩 이동하며 **빠져나가는 맨 앞의 초밥 개수는 빼고, 새로 들어오는 맨 뒤의 초밥 개수는 더하는 방식**으로 변경했습니다. 이로써 불필요한 반복을 없애고 시간 복잡도를 O(N)으로 단축했습니다.
- **모듈러(%) 연산 활용:** 새로 들어오는 초밥의 인덱스를 계산할 때 `(i + k) % N`을 사용하여, 인덱스가 배열의 범위를 벗어나면 자연스럽게 `0`번 인덱스로 순환되도록 원형 배열 문제를 해결했습니다.

**💡 새롭게 알게 된 점**

- **슬라이딩 윈도우와 상태 추적:** 윈도우 내 요소들의 상태를 추적할 때 단순히 존재 유무만 중요하다면 `Set`이 좋지만, '개수'의 증감이 윈도우의 상태(초밥 가짓수)에 영향을 미칠 때는 **카운팅 배열**이나 해시맵(Map)을 써야 한다는 것을 깨달았습니다.
- **원형 배열 테크닉:** 배열의 끝과 처음이 이어진 구조를 탐색할 때, 배열을 물리적으로 2배 늘려서 이어 붙이는 방법도 있지만, 나머지 연산자(`%`)를 사용하면 메모리 낭비 없이 아주 우아하게 인덱스를 순환시킬 수 있다는 점을 배웠습니다.

**🛠 아쉬운 점 & 리팩토링 계획**

- **현재 상태:** 제공된 최종 코드는 JS 환경에서 메모리와 속도에 유리한 `Int32Array`까지 적용하여 이미 상당히 최적화된 상태입니다.
- **확장성 고려:** 현재 문제는 초밥의 가짓수(`d`)가 최대 3,000개라 고정 길이 배열로 충분히 커버가 가능했습니다. 하지만 만약 문제 조건이 바뀌어 초밥의 가짓수(`d`)가 10억 개처럼 매우 크고, 레일 위의 초밥 종류는 듬성듬성하다면 메모리 낭비가 심해질 것입니다. 이럴 때는 배열 대신 **`Map` 객체**를 사용하여 들어오는 초밥의 번호만 키(Key)로 등록하는 방식으로 리팩토링해 볼 수 있겠습니다.
